# 1. Мета

Побудувати **production-ready систему моніторингу ресурсів KDI** на базі **VictoriaMetrics + Grafana**, яка:

- дає повний контроль над станом продакшну;
- рано виявляє деградацію та вузькі місця;
- підтримує подальший апскейл і підключення нових систем без хаосу;
- створює основу для capacity planning і стабільного розвитку KDI.

---

## 2. Контекст

KDI складається з критичних сервісів Koha, DSpace, Integrator та їх залежностей. Поточний базовий monitoring-рівень недостатній для production: бракує глибоких метрик по хостах, контейнерах, БД, пошуку, чергах і прикладних сервісах. Через це складно:

- швидко знаходити причину деградації;
- приймати рішення щодо оптимізації;
- безпечно масштабувати систему;
- підключати нові сервіси у стандартизований спосіб.

---

## 3. Product Vision

Observability у KDI має бути не набором випадкових графіків, а **окремим платформним шаром**, який:

- централізовано збирає метрики з усіх критичних компонентів;
- дає єдиний інтерфейс перегляду стану системи;
- підтримує алертинг для production-інцидентів;
- масштабується разом із системою;
- має стандарти для підключення нових сервісів, хостів і мікросервісів.

---

## 4. Основні цілі

### Бізнес-цілі

- зменшити час виявлення інцидентів;
- зменшити ризики деградації продакшну;
- забезпечити кероване масштабування KDI;
- дати фактичну базу для performance tuning і планування ресурсів.

### Технічні цілі

- централізований збір метрик з усіх критичних компонентів;
- історичне зберігання метрик для аналізу трендів;
- production alerting для основних ризиків;
- єдиний стандарт labels, dashboards і підключення нових систем.

---

## 5. Що входить у scope

### Обов’язково для production

- **VictoriaMetrics** як централізоване сховище метрик;
- **Grafana** як єдиний інтерфейс дашбордів;
- **vmagent** як єдиний шар збору і доставки метрик;
- моніторинг **Linux hosts**;
- моніторинг **Docker / containers**;
- моніторинг **Traefik**;
- моніторинг **PostgreSQL**;
- моніторинг **MariaDB**;
- моніторинг **RabbitMQ**;
- моніторинг **Memcached**;
- моніторинг **Elasticsearch / Solr**;
- моніторинг **Integrator**;
- критичний alerting;
- базові capacity dashboards;
- стандарт підключення нових систем.

### Що не є пріоритетом на старті

- централізовані логи;
- distributed tracing;
- глибокі бізнес-метрики;
- складні executive dashboards;
- надмірна деталізація app-level метрик, якщо вона не потрібна для стабільності продакшну.

## 6. Ключові користувачі

### DevOps / Platform engineer

Потребує повної картини хостів, контейнерів, БД, проксі, черг і алертів.

### Розробник / власник сервісу

Потребує метрик для Integrator та інших сервісів, щоб бачити латентність, помилки, навантаження і наслідки змін.

### Архітектор / техлід

Потребує трендів, вузьких місць і capacity view для прийняття рішень щодо масштабування.

---

### Інфраструктурний рівень (Host & Docker)

- **Node Exporter:** Збір базових метрик хост-машини (навантаження CPU, використання RAM, I/O дисків `VOL_ASSETSTORE_PATH`, мережевий трафік).
- **cAdvisor:** Збір метрик на рівні кожного окремого Docker-контейнера (kohanet, dspacenet).
- **Traefik Metrics:** Активація вбудованого Prometheus-ендпоінту в Traefik v3.6 для відстеження кількості HTTP-запитів, кодів відповідей (4xx, 5xx), та часу відгуку (Latency) маршрутизації до DSpace та Koha.

### 3.2. Бази даних та Сховища (Data Layer)

- **MariaDB (Koha):** Розгортання `mysqld-exporter`. Ціль: моніторинг з'єднань, кешу InnoDB для подальшого Performance Tuning.
- **PostgreSQL 15 (DSpace):** Розгортання `postgres-exporter`. Ціль: моніторинг транзакцій, розміру таблиць, мертвих рядків (dead tuples).
- **Memcached 1.6:** Розгортання `memcached-exporter`. Ціль: відстеження Cache Hit/Miss ratio для налаштування жорсткого хіт-рейту.
- **RabbitMQ 3.8+:** Активація вбудованого `rabbitmq_prometheus` плагіна. Ціль: моніторинг розміру черг демона автоіндексації (`koha-es-indexer`).

### 3.3. Пошукові рушії (Search Layer)

- **Elasticsearch 8.19.6:** Розгортання `elasticsearch-exporter`. Ціль: моніторинг стану кластера (Yellow/Red/Green), розміру індексів.
- **Apache Solr 9.8:** Активація збору JMX-метрик або Solr Prometheus Exporter.

### 3.4. Рівень Застосунків (Application Layer)

- **Koha (Apache2 + Plack):** Розгортання `apache-exporter`. Тюнінг кількості воркерів Plack на основі навантаження OPAC (порт 8080) та Intranet (8081).
- **DSpace (Tomcat / Spring Boot 3):** Інтеграція `jmx-exporter` або активація Spring Boot Actuator (`/actuator/prometheus`) для аналізу збірки сміття (GC) та пулу потоків Java.
- **KDV Integrator (Python/Flask):** Інтеграція бібліотеки `prometheus_client` у код Flask для виводу метрик черги In-Memory `TASKS` (кількість queued, processing, failed) до моменту міграції на Redis.

## 7. Ключові функціональні вимоги

## FR-1. Єдиний metrics backend

Уся система повинна використовувати **VictoriaMetrics** як централізоване сховище метрик.

## FR-2. Єдиний шар збору

Усі exporter-и та metrics endpoints повинні збиратись через **vmagent**, щоб нові системи підключались за єдиною схемою.

## FR-3. Обов’язкове production-покриття

Система повинна покрити метриками критичні рівні:

- host;
- container;
- reverse proxy;
- databases;
- queue/cache/search layer;
- Integrator.

## FR-4. Мінімально необхідні dashboards

Мають бути щонайменше такі дашборди:

1. **KDI Global Health**
2. **Hosts & Docker**
3. **Databases**
4. **Search & Queue Layer**
5. **Integrator Overview**
6. **Capacity Planning**

## FR-5. Critical alerting

Мають бути налаштовані алерти з відправкою повідомлень на пошту MS365 щонайменше на:

- host down;
- disk almost full;
- memory pressure;
- CPU saturation;
- container restart storm;
- PostgreSQL/MariaDB degradation;
- RabbitMQ queue growth;
- Memcached degradation;
- Elasticsearch/Solr unhealthy;
- Traefik 5xx spike;
- Integrator latency/error spike;
- observability stack unhealthy.

## FR-6. Scale-ready design

Архітектура повинна дозволяти безболісно підключати:

- нові хости;
- нові Docker Compose стеки;
- нові БД;
- нові мікросервіси;
- нові зовнішні системи.

Для цього обов’язкові:

- стандарт labels;
- шаблон підключення нового сервісу;
- правила проти cardinality explosion;
- version-controlled scrape/config rules.

## 8. Нефункціональні вимоги

### Надійність

Відмова observability stack не повинна валити production.

### Безпека

 Усі метрики (`/metrics`) повинні бути закриті від зовнішнього світу. `vmagent` збирає їх виключно всередині Docker-мереж (`dspacenet`, `kohanet`). Інтерфейс Grafana має бути доступний або через захищений Cloudflare Tunnel (із Cloudflare Access),.

### Низький overhead

 База даних метрик не повинна забирати ресурси в основних застосунків. VictoriaMetrics (Single-node) має бути обмежена лімітами Docker (наприклад, макс. 1-2GB RAM).

### Керованість

Конфіги дашбордів, scrape targets і alert rules повинні бути керовані через Git і стандартизовані.

### Масштабованість

Архітектура повинна підтримувати подальше зростання кількості джерел метрик і обсягу історичних даних.

**Персистентність даних:** Дані VictoriaMetrics мають зберігатися у підключеному томі Docker (Docker Volume), налаштованому у `.env` (Single Source of Truth).

**Зберігання (Retention):** Дані метрик мають зберігатися мінімум 1 рік для забезпечення аналітики "рік до року" без суттєвої деградації швидкості читання.

---

## Архітектура збору (Push Model)

Замість того, щоб центральна база йшла до кожного контейнера, ми використовуємо `vmagent`:

1. У кожен docker-compose файл (Koha, DSpace) додається легковигий контейнер `vmagent` та необхідні `exporter`и.
2. `vmagent` локально скрапить (збирає) метрики зі своїх сусідніх контейнерів.
3. `vmagent` відправляє їх (Remote Write) у центральний контейнер `victoriametrics`, який може знаходитись у мережі моніторингу.

## 9. Модель відповідальності

### DevOps / Platform layer

Відповідає за:

- VictoriaMetrics;
- Grafana;
- vmagent;
- node/container/database/platform exporter-и;
- scrape configs;
- labels;
- базові dashboards;
- alerting.

### Власник конкретного сервісу

Відповідає за application-level метрики свого сервісу, якщо потрібне глибше інструментування.

### Архітектор / техлід

Відповідає за стандарти моніторингу, правила масштабування, пріоритети покриття і governance.

## 10. Success Criteria

PRD вважається реалізованим, якщо:

- усі критичні production-компоненти KDI мають metrics coverage;
- VictoriaMetrics, Grafana та vmagent розгорнуті як Docker-контейнери без конфліктів з існуючими стеками.
- Налаштовано автоматичний бекап конфігурацій Grafana (Dashboards & Data Sources) через механізм Provisioning, щоб уникнути втрати налаштувань при перестворенні контейнерів.
- Проведено первинний аналіз метрик MariaDB та Memcached, на основі якого внесено зміни у `.env` (live patching) для усунення Performance Gaps.
- **Успішно налаштовано доставку Email-алертів (через MS365 SMTP), перевірено доставку тестового сповіщення на пошту адміністраторів системи.**
- усі production hosts і containers видно в Grafana;
- існують мінімальні production dashboards;
- critical alerts спрацьовують до скарг користувачів;
- можна побачити ресурсні bottleneck-и й тренди росту;
- нову систему можна підключити до моніторингу за стандартним шаблоном без переробки архітектури.

---

## 11. Найважливіші ризики

- хаотичне підключення exporter-ів без стандарту;
- зайва експозиція metrics endpoints;
- alert fatigue через погано налаштовані правила;
- cardinality explosion;
- сліпі зони на рівні application metrics;
- часткове покриття лише хостів без БД і сервісних залежностей.

---

## 12. Підсумок

Для production і подальшого апскейлу KDI Observability повинна будуватись навколо шести ключових принципів:

1. **єдиний metrics backend**;
2. **покриття всіх критичних компонентів**;
3. **мінімальний, але сильний набір дашбордів**;
4. **critical alerting**;
5. **стандарти для підключення нових систем**;
6. **capacity planning як частина архітектури, а не додаткова опція**.

Саме цього достатньо, щоб зробити систему придатною до стабільного production-використання і контрольованого росту.
